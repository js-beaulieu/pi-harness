# Development loop

The lifecycle is `plan → code → review → done`; only `/workspace:code` and `/workspace:done` accept user-controlled transitions. The parent orchestrates and never edits `projects/**`. This skill and `workspace-knowledge-protocol` are the workflow authority. Do not inspect Pi package files, `node_modules/`, or `.pi/` internals to work out the phases.

## Plan

Begin from the primary coordination root with `/workspace:plan <subject>`. It creates one isolated worktree containing both the coordination checkout and configured product checkouts, runs manifest setup hooks directly from the extension, forks the current Pi conversation, and switches the active runtime into that cwd. Each hook has its new worktree as `$PWD`; `PH_WORKSPACE_BASE` points to the primary coordination root, and project hooks also receive `PH_PROJECT_BASE`. Use `/workspace:continue <id>` to return to a saved workflow; do not create an ad-hoc product-only worktree set.

Before direct code reading: (1) call `workspace_knowledge(action="orientation")`, then search/read the relevant project Knowledge map and its narrowest relevant capability/component nodes, plus related architecture, contracts, and decisions; (2) inspect and call the available `codebase-memory` MCP tools for structural discovery. Direct reads of non-doc files are then for targeted evidence only. Create durable Plan and Task entries with `workspace_knowledge_record`; give it factual named sections instead of hand-formatting Markdown. Identify projects, dependency order, contracts, risks, validation, and acceptance criteria. Create or materially update a `Reference` titled `Project architecture — <project>` for every affected project: purpose, owned responsibilities, entry points, key modules, data flow, external interfaces, configuration, validation, and useful code-graph anchors. This is durable orientation, not a task journal. Create branches only; do not edit product code.

After orientation and graph evidence, choose the branch name yourself. Use an appropriate work-type prefix (`feat`, `fix`, `docs`, `refactor`, `test`, `ci`, `build`, `chore`, `perf`, or `style`) plus the shortest clear description of the actual delivered outcome; do not mechanically copy or slug the user’s wording. This convention may mirror Conventional Commit types, but it is not governed by the Conventional Commits specification. Create and record the same concise branch in the workspace and affected product repositories before writing knowledge or code. The package’s suggested branch is only a fallback when evidence does not yield a better name.

The parent may inspect non-doc workspace and product files. `docs/` is never read directly: use `workspace_knowledge` exclusively. For a requested historical sweep, follow the backfill protocol in `workspace-knowledge-protocol`; use history only as evidence for what/why/how/where knowledge-tree claims; do not create ad hoc `git log` summaries. If the graph is unavailable, record the exact failure in the Task and use targeted reads as a fallback; never pretend graph evidence exists. For every task that is not narrow and graph-known, launch `explore` before forming the implementation handoff. The parent may synthesize known graph/knowledge facts directly only for a genuinely narrow task; it must not replace exploration with broad personal file spelunking. When work changes structure, ownership, interfaces, configuration, or operations, update the relevant Project Architecture Reference, Contract, or Decision before review; record the delivered outcome and the reason it matters, not a narration of agent activity.

## Code

Launch workers only via the `subagent` tool, with the agent name exactly `impl` or `impl-lite`; never use a generic `delegate` agent or a foreground parallel run. Route one-file documentation, test, config, and localized implementation changes to `impl-lite`; reserve `impl` for multi-file, cross-project, contract, or genuinely complex work. Use `async: true`, `clarify: false`, `context: "fresh"`, `artifacts: false`, `acceptance: { level: "none", reason: "Pi Harness owns worker evidence, CI, and review." }`, and no `cwd` override—the child must start at the coordination root to load the Harness extension and tool set. `artifacts: false` is required: session/status output remains available, but disposable Pi Subagent files must never land in product repositories. Pi Subagents' inferred acceptance is deliberately disabled: it duplicates the Harness gate, wastes context on JSON boilerplate, and rejects legitimate no-change work. State the target checkout path in the task. Every delegation includes: target `AGENTS.md`, graph evidence, acceptance criteria, checkout, branch, and configured CI; do not direct a worker to read canonical docs or call workspace knowledge. Continue independent orchestration work; if waiting is necessary, use `wait`, never a status-polling loop. Workers return changed files, commands/results, risks, and blockers. The parent runs or collects configured CI, records checks, updates knowledge, and opens PRs where available. After implementation stabilizes and before review, obtain `workspace_knowledge(action="session_scope")` for each changed project. Use the code graph to identify changed symbols/interfaces, call `workspace_knowledge_impact`, and disposition every affected node and unmapped path before calling `workspace_knowledge_tree_record`. Record only material what/why/how/where claims and locators established by this session; no final or production commit ID is required, and review is blocked until every changed project completes the checklist.

## Review and done

Before review, commit the workspace’s `docs/`, `workspace.yaml`, and other tracked coordination changes on its own workflow branch. If the coordination repository has a GitHub origin and local `gh` push access, create and record its PR as `workspace`; it is reviewed alongside product PRs. Enter review only after CI and graph evidence are recorded. A PR is required only for an affected project—or the workspace itself—with a GitHub origin for which the local `gh` session confirms push/PR capability; local-only, non-GitHub, or inaccessible remotes do not block review. Review is read-only. `/workspace:done` summarizes available PRs in dependency order; never merge or force-push.
