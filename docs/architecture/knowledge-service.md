# Knowledge service

`docs/` is the canonical, Git-reviewed knowledge base. It is not an opaque memory store and it is never generated from a code graph.

The local Pi knowledge tool provides a durable, Git-reviewed orientation layer:

- `Decision` — an accepted choice, alternatives, consequences, and review date.
- `Plan` — scope, acceptance criteria, dependencies, and validation plan.
- `Task` — lifecycle state, current step, blockers, and resumption context.
- `Contract` — owned interface, consumers, compatibility policy, and validation.
- `Reference` — durable orientation or operational knowledge; every actively maintained product has a `Project architecture — <project>` reference.

Entries are Markdown with an H1 and factual sections. Well-known kinds receive useful starter headings, but kinds and headings remain flexible so the tool records knowledge rather than rejecting valid facts for formatting reasons. `workspace_knowledge` exposes orientation over project Knowledge maps, durable References, Contracts, and Decisions; `workspace_knowledge_record` updates normal named sections, while `workspace_knowledge_tree_record` maintains generated tree nodes for both backfill and current-session work. `workspace-knowledge-protocol` defines what belongs in each entry and what does not.

Only the orchestrator may access or write canonical knowledge. Implementation and exploration subagents report product-checkout findings back; the orchestrator turns material reports, decisions, deviations, and contract changes into durable documentation. Writes are allowed in `plan` and `code` only, and must be complete before review.

Updates are section-scoped, not document rewrites. Normalized no-ops do not write the file or update its timestamp; wording-only and formatting-only changes are prohibited by protocol rather than guessed at by a model. Plans and Tasks capture workflow state. Project maps and capability/component nodes steer later agents to the smallest useful graph or source query; References, Contracts, and Decisions provide broader architecture and constraints.

## History-assisted knowledge-tree construction

Git history is analysis evidence, not a user-facing knowledge category. `workspace_knowledge(history_status)` reports remaining history, tracked files, bytes, text lines, and whether the current snapshot is validated. The tool infers low, moderate, or high structural complexity from repository/history scale, so planning requires no graph, shell, source reads, or permission prompts. `history_plan` derives a contiguous oldest-first segment layout from history length, repository size, complexity, and a bounded evidence budget; there is no fixed commit count. `history_start` combines extension-held plans by project name—without model-carried plan/run tokens—into one confirmation gate before evidence is released. The complete plan includes commit subjects, unique changed files, added/deleted lines, evidence bytes, and repository scale. Users may split a segment or merge adjacent segments by replacing contiguous segment sizes; commit chronology cannot change. `history_chunk` releases only the next segment, and that segment must be recorded before progress advances. Bodies are capped at 8,000 characters, paths at 200 entries, and patches at 6,000 characters per change. Initial coverage includes all commits reachable from the snapshot; later coverage uses `last-covered..new-snapshot` without repartitioning reviewed evidence.

A dedicated read-only `backfill` agent analyzes one chronological segment at a time using project reads, read-only Git commands, and the code graph. It cannot write product code or canonical docs. The orchestrator converts its evidence into durable claims under stable node paths (`architecture`, `features`, `interfaces`, `data`, `operations`, or `quality`). Each node has impact anchors: path patterns for broad ownership, qualified path/name/kind symbols for precision, interface identifiers, and related knowledge nodes. Claims have stable keys, one of `What`, `Why`, `How`, or `Where`, concise facts, and source/search locators.

Historical messages may help the agent recover rationale, but generated knowledge records only the decision or intent—not the Git record used to discover it. Changed paths establish relevance and location but cannot establish intent. Chronology and change narration are discarded. A scope with no durable findings is processed with no nodes.

`workspace_knowledge_impact` forms a reverse impact set from changed paths, changed symbols supplied after the code-graph check, changed interface identifiers, and one-hop knowledge relations. Relation expansion deliberately stops after one hop so a connected tree does not turn every local change into a whole-project review. It also reports changed paths owned by no node. Its input accepts exactly `graph.status=checked` or `graph.status=unavailable`; unavailable requires a reason. Every existing node must be marked `updated`, `unchanged`, `moved`, or `retired`, and every uncovered file must be classified `new-node` or `no-durable-knowledge`. Updated/moved/retired dispositions require a replacement node, unchanged and no-knowledge dispositions require reasons, and new-node classifications require a supplied node whose path anchors cover the change.

The tools exchange opaque scope and impact tokens so the model never reconstructs hashed internal objects or depends on JSON key ordering. `workspace_knowledge_tree_record` accepts an impact token for a chronological history segment, a final current snapshot, or a `workspace:code` session scope. Session scope is computed from the active workflow and project changes relative to its configured default branch, so it does not depend on eventual production commit IDs. In both modes, stable claim keys update independently, locators must intersect changed paths, and unchanged locator state prevents rewording. Intermediate root maps show an explicit incomplete warning with reviewed/total commits. Finishing history changes the warning to current-snapshot-validation pending rather than claiming readiness. `current_scope` then covers the full current tree: impact analysis reviews every existing node and reports every unanchored tracked file; code files cannot receive a no-knowledge disposition. Updated nodes provide a complete current claim set, allowing obsolete claims to disappear, while retired nodes are removed from readable maps and documents. The recorder verifies that every current code file is anchored by an active node and that active locators resolve before marking the map current-snapshot validated. It also generates a code-file index that names every current code path and links it to its owning guidance node, eliminating silent code orphans while keeping the capability tree concise. Internal state prevents replay, while unrelated scopes do not regenerate readable nodes. `enter_review` independently recomputes each changed project's current session state and refuses review until its checklist was recorded.

Generated Markdown nodes contain `What`, `Why`, `How`, and `Where` sections with location hints and no change-history narration. Active final-snapshot nodes require current What, How, and Where guidance so a future orchestrator understands the responsibility, implementation flow, and best investigation starting point. Deterministic `README.md` maps at the project and category levels expose child summaries and a few search anchors. Orientation returns only project roots; the agent progressively opens a category and then the narrowest node before using the code graph or source. The tree complements rather than duplicates Git and the code graph.

## Enforcement and future MCP interface

The custom Pi tool is phase-aware because it runs in the same process as the workflow extension. It is the write interface used by the initial workflow.

The next extraction point is a local stdio MCP server with the same operations. It will read the same Markdown files and maintain only a disposable local search index. Pi will consume it through `pi-mcp-adapter`; other harnesses can use its standard `.mcp.json` entry. Before exposing write tools through the adapter, the server will require a short-lived workflow capability issued by the extension, so an arbitrary MCP caller cannot bypass phase gates.

That capability bridge is intentionally not faked in this initial package: a generic MCP proxy has no reliable Pi-session identity on its own. The in-process tool keeps the enforcement claim true today.
